## Decisiones técnicas tomadas

### 1. Campo opcional en edición de tasas
**Decisión:** `boss_commission_default` es OPCIONAL en edit, REQUERIDO en create.

**Razón:**
- Create: siempre establecer comisión inicial
- Edit: permite actualizar solo tasas sin forzar cambio de comisiones
- Más flexible y explícito para el usuario

**Implementación:**
```php
// Create
'boss_commission_default' => 'required|numeric|min:0|max:100'

// Edit
'boss_commission_default' => 'nullable|numeric|min:0|max:100'

// Solo actualiza si fue enviado
if ($request->filled('boss_commission_default')) {
    Seller::query()->update(['boss_commission' => $value]);
}
```

### 2. No usar promedio de comisiones
**Decisión:** No usar `avg('boss_commission')` como valor por defecto.

**Razón:**
- El promedio no representa ninguna "comisión global real"
- Si hay overrides, el promedio pierde significado
- Mejor mostrar distribución completa

**Implementación:**
```php
// Mostrar distribución, no promedio
$commissionGroups = Seller::select('boss_commission')
    ->groupBy('boss_commission')
    ->selectRaw('boss_commission, count(*) as count')
    ->orderBy('count', 'desc')
    ->get();
```

### 3. Snapshots de tasas agregados
**Decisión:** Agregar snapshots de tasas (USD, EUR, VES) además de comisiones.

**Razón:**
- Trazabilidad completa para auditoría
- Permite saber qué tasas se usaron en cada venta
- Útil para reportes históricos y reconciliación

### 4. Comisión NO se guarda en exchange_rates
**Decisión:** La comisión aparece en formulario de tasas pero NO se guarda en la tabla.

**Razón:**
- Evita redundancia (ya está en cada seller)
- Permite overrides por vendedor
- Formulario es solo interfaz centralizada
- Actualización masiva directa en `sellers`

### 5. REQ 4 sin refactoring innecesario
**Decisión:** Verificar y documentar en lugar de reimplementar.

**Razón:**
- Sistema ya funciona correctamente
- Ahorra tiempo (~19h)
- Evita introducir bugs
- Mejor usar tiempo en REQ pendientes
