## Decisiones técnicas tomadas

### 1. Código único con formato VEN-XXXXXX
- **Razón:** Legible, fácil de comunicar, único
- **Generación:** En `boot()` del modelo para garantizar unicidad
- **Probabilidad colisión:** <0.001% (36^6 combinaciones)

### 2. Motor de cálculo con snapshots
- **Razón:** Usar snapshots de comisiones guardados en sales
- **Beneficio:** Precisión histórica incluso si cambian comisiones
- **Fallback:** Calcula dinámicamente si no hay snapshot

### 3. Monedero con balance_after en cada transacción
- **Razón:** Auditoría completa + performance O(1)
- **Beneficio:** Saldo actual = última transacción
- **Trazabilidad:** Detecta inconsistencias si se modifica BD

### 4. Automatización de comisiones en Observer
- **Razón:** Centralización de lógica de cambios de estado
- **Beneficio:** Funciona sin importar cómo se apruebe (API, Tinker, CLI)
- **Trigger:** Venta pasa a approved/completed → +comisión

### 5. UI/UX con gradientes animados
- **Razón:** Impacto visual para video demo
- **Implementación:** CSS animations + Tailwind
- **Paleta:** Morado → Rosa → Turquesa (Cambio J)

### 6. Liquidaciones con DB::transaction
- **Razón:** Atomicidad (o se crea todo o nada)
- **Beneficio:** Evita liquidation sin wallet_transaction
- **Validación:** Saldo suficiente antes de guardar
